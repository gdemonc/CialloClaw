# Shell-Ball 桌面悬浮球化补充设计

## 1. 文档目的

本文用于补充 `shell-ball` 第一阶段形态系统之后的下一步方向约束，明确其未来应收敛为“桌面悬浮球入口”，而不是长期保留为网页式展示页面。

本文同时整理以下新增共识：

- `shell-ball` 不应继续保留完整 `DemoPage` 形态
- `shell-ball` 的交互触发逻辑应更稳，不应过度敏感
- 当前 7 个冻结展示态继续保留，不新增新的对外展示态
- 上方气泡区后续单独实现；本轮仍只保留预留结构
- 下方交互区继续限制为单一 input bar，不恢复页面式承接面板

## 2. 当前共识

### 2.1 产品定位

`Shell-Ball` 的目标不是“在网页里展示一个悬浮球”，而是未来作为桌面上的常驻悬浮入口存在。

因此，后续设计应以“桌面悬浮球窗口”作为主形态，而不是以“网页演示页”作为主形态。

### 2.2 当前必须继续保留的边界

- 冻结的 7 个展示态不变：
  - `idle`
  - `hover_input`
  - `confirming_intent`
  - `processing`
  - `waiting_auth`
  - `voice_listening`
  - `voice_locked`
- 上方气泡区仍然只做预留，不在本轮展开完整气泡系统
- 下方交互区仍然只保留：
  - 输入框
  - 附加文件按钮
  - 发送按钮
- `confirming_intent`、`waiting_auth` 继续保持自动过渡
- 仍然不接入真实文件拖拽、真实文本选中、真实语音识别、真实 RPC 主链路

## 3. 对当前实现的判断

### 3.1 已有独立窗口入口，但 UI 结构仍偏“网页页”

当前仓库中已经存在 `shell-ball` 独立窗口入口：

- `apps/desktop/src-tauri/tauri.conf.json`
- `apps/desktop/shell-ball.html`
- `apps/desktop/src/app/shell-ball/main.tsx`

且窗口已经具备如下方向：

- `decorations: false`
- `transparent: true`

这说明 `shell-ball` 作为独立桌面窗口的宿主入口已经存在。

### 3.2 当前真正的问题

当前问题不在于“没有独立窗口”，而在于 `ShellBallApp` 仍承担了较强的页面展示职责，例如：

- 页面标题
- 页面说明文案
- 大页面外框
- 调试切换器作为页面主体的一部分

这些内容适合开发期演示，但不适合作为长期桌面悬浮球的正式结构。

## 4. 结构调整方向

### 4.1 不再保留长期 `DemoPage`

后续不建议继续维护单独的“网页展示页”概念。

更合理的方向是：

- 保留 `shell-ball.html` 作为桌面窗口入口
- 将 `ShellBall` 主体收敛成一个纯悬浮球表面组件
- 将开发期演示能力收敛成 dev-only 调试能力，而不是页面主体结构

### 4.2 推荐结构

#### `ShellBallSurface`

职责：

- 纯悬浮球核心表面
- 只包含以下内容：
  - 上方气泡区插槽
  - 鸟球本体
  - 下方 input bar

不负责：

- 页面标题
- 页面说明
- 大页面容器
- 演示说明页结构

#### `ShellBallDesktopWindow`

职责：

- 面向桌面悬浮球窗口的外层容器
- 负责：
  - 窗口透明背景适配
  - 窗口定位
  - 后续始终置顶
  - 后续拖拽/吸边
  - 桌面模式下的边距与交互区控制

当前策略：

- 该层明确与 Tauri 窗口能力相关
- 但不应早于 `ShellBallSurface` 稳定化而提前接入
- 只有在纯前端表面结构、交互节奏和 dev-only 调试层都满意后，才进入该层开发

#### dev-only 调试层

当前 `ShellBallDemoSwitcher` 不应继续作为页面主体存在。

后续建议：

- 仅在开发环境下显示
- 作为隐藏调试层或浮层存在
- 不进入正式桌面悬浮球主视觉结构

## 5. 交互稳定性调整原则

当前 `shell-ball` 的交互触发偏敏感，后续应统一收敛为“更稳、更克制”的触发策略。

### 5.1 Hover 触发

`idle -> hover_input` 不应在鼠标划过时立刻触发。

建议改为：

- `hover intent` 延迟约 `240ms`
- 离开缓冲约 `180ms`

补充规则：

- 如果输入框已聚焦，不自动立即收起
- 如果用户已输入草稿，不自动立即收起
- 鼠标短暂穿过交互区，不应触发完整展开

### 5.2 语音触发

`voice_listening` 不应在短按或轻触下过快触发。

建议改为：

- 长按约 `420ms` 后进入 `voice_listening`
- 长按前的小幅移动不解释为语音手势
- 进入语音后，先设置手势死区，避免抖动误判

### 5.3 锁定 / 取消逻辑

后续语音手势改为：

- 上拖越过阈值后，只进入“预览”
- 下拖越过阈值后，只进入“预览”
- `松手后提交`

即：

- 不再采用“越过阈值立即锁定/取消”
- 统一改为“越过阈值预览，松手确认”

建议阈值：

- 上拖锁定阈值：`48px`
- 下拖取消阈值：`48px`

## 6. 与未来拖拽能力的兼容原则

后续 `shell-ball` 明确需要支持拖拽，因此当前交互必须为此预留空间。

### 6.1 手势识别优先级

建议遵循：

1. 先识别是否达到长按阈值
2. 再识别是否进入语音手势阶段
3. 再判断是否为明显纵向动作
4. 非明显纵向动作先忽略或保留给未来拖拽能力

### 6.2 兼容策略

- 小幅移动不应触发锁定/取消
- 水平移动不应过早被解释为语音锁定
- 语音锁定采用“松手提交”后，未来更容易与拖拽行为解耦

## 7. 后续推荐实现顺序

当前推荐策略：先把纯前端 `shell-ball` 表面做稳，再做 Tauri 相关宿主适配。也就是说，先完成视觉表面、交互节奏、调试层收敛与拖拽兼容边界，再进入窗口层。

### 7.1 第一优先级：从页面壳收敛为桌面表面

- 将当前 `ShellBallApp` 收敛为 `ShellBallSurface`
- 去掉页面标题、说明文案、大页面展示框
- 保留核心三段结构：
  - 气泡区预留
  - 鸟球主体
  - input bar

### 7.2 第二优先级：调试能力收敛为 dev-only

- `ShellBallDemoSwitcher` 改为仅开发态可见
- 不再作为正式结构的一部分
- 不再依赖完整 demo page 承载

### 7.3 第三优先级：交互阈值稳定化

- 增加 hover intent
- 增加 leave grace
- 提高长按阈值
- 将锁定/取消改为“松手提交”
- 增加手势死区与纵向优先判定

### 7.4 第四优先级：为未来拖拽与宿主能力预留边界

- 明确哪些交互属于纯前端表面层
- 明确哪些能力属于未来窗口宿主层
- 保持手势与拖拽的边界清晰，避免后续 Tauri 适配时返工

### 7.5 第五优先级（后置）：窗口能力正式接入

后续通过平台抽象层补充：

- show / hide
- focus
- reposition
- always-on-top
- 桌面拖拽
- 吸边 / 贴边 / 停靠

不得把这些平台行为直接写进 `shell-ball` 业务展示组件内。

进入本阶段的前提是：

- `ShellBallSurface` 已稳定
- 页面展示壳已完全移除
- dev-only 调试层已收敛
- 交互节奏与拖拽兼容边界已满意

## 8. 本轮不做的事

本次只整理方向，不在本轮直接展开：

- 完整气泡系统
- 真实文件拖拽承接
- 真实文本选中承接
- 真实语音识别
- dashboard 联动
- `ShellBallDesktopWindow` / Tauri 宿主适配
- 桌面吸边/贴边
- 完整桌宠行为状态机

## 9. 一句话结论

`Shell-Ball` 后续应继续沿“桌面悬浮球窗口”方向收敛：保留 7 态和核心交互控制器，移除网页式展示壳，降低触发敏感度，并为未来拖拽和桌面窗口行为预留清晰边界。
