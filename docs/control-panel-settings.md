# 控制面板设置文档

## 文档定位

本文用于定义 **控制面板设置** 需要承载的配置项、信息组织方式、用户可感知的状态以及关键交互规则，目的是让控制面板成为桌面端 Agent 的统一设置入口，而不是零散开关的堆叠。

本文重点回答以下问题：

- 用户可以配置哪些内容，这些内容分别影响什么。
- 设置项应如何分组，用户如何快速理解每一组设置的作用范围。
- 哪些设置即时生效，哪些设置需要重启、重新加载或再次确认。
- 涉及记忆、自动化、下载、本地数据与成本的设置，如何保证可解释、可控、可恢复。

本文不展开具体技术实现，仅约束设置内容、页面组织、状态表达与交互要求。

---

## 1. 文档范围与设计原则

控制面板设置不是“参数仓库”，它应该承担三层职责：

1. **基础控制层**：允许用户快速调整语言、外观、通知、悬浮球等高频基础设置。
2. **协作策略层**：允许用户配置记忆保留周期、便签巡检规则、提醒与自动化行为，让 Agent 的协作方式符合个人偏好。
3. **边界与成本层**：允许用户理解并管理模型、Token、预算、日志、本地存储与隐私策略，建立长期信任。

因此，设置页的组织应遵循以下原则：

### 1.1 可理解

每一项设置都应告诉用户：

- 这项设置控制什么。
- 开启或关闭后会带来什么变化。
- 是否会影响仪表盘、控制面板、任务执行或镜子记忆。

### 1.2 可预期

涉及重启、生效延迟、模型切换、巡检频率变化、记忆删除、恢复默认等动作时，必须明确说明：

- 是否立即生效。
- 是否影响当前任务。
- 是否可撤回。

### 1.3 可追溯

与记忆、任务历史、日志、预算相关的设置，不应只提供开关，还应提供查看入口、影响说明和清理动作。

### 1.4 分层清晰

高频基础设置放前，低频风险设置放后；视觉与体验类设置、协作机制类设置、数据治理类设置要分区明确，避免“同一层级混放”。

### 1.5 与仪表盘语言统一

设置页中的文案应与仪表盘、任务状态、镜子、安全卫士使用同一套状态语言，避免用户在不同页面看到不同说法。

## 2. 信息架构总览

### 2.1 当前正式导航

当前桌面实现中的控制面板已冻结为以下 **6 个正式一级分组**：

1. **通用设置**：语言、开机自启、主题、声音与下载设置。
2. **悬浮球**：悬浮球在场方式、透明度、停靠模式，以及暂未开放写回的贴边/尺寸快照。
3. **记忆**：镜子记忆开关、保留周期与总结节奏。
4. **任务与巡检**：任务来源、巡检频率、启动巡检、文件变化巡检与提醒策略。
5. **模型与安全**：Provider、Base URL、模型名、API Key 状态、预算自动降级，以及安全摘要与成本摘要。
6. **关于**：帮助与反馈、分享入口，以及当前桌面端版本信息。

补充说明：

- 控制面板保存时会拆分设置快照与巡检配置，只提交发生变化的分组，不额外触发无关配置写入。

### 2.2 预留展示分组

以下展示内容当前尚未在桌面控制面板中冻结为独立导航分组，但应继续保留在设计稿中，供后续能力补齐时直接对齐：

1. **数据与日志**：本地存储位置、数据清理、日志查看、预算与历史成本治理。

页面结构建议采用“左侧导航 + 右侧详情”的标准设置页布局：

- 左侧：一级设置分组。
- 右侧顶部：当前分组标题与必要状态提示。
- 右侧内容区：按卡片分块展示设置项。
- 页面底部：全局操作区，例如“恢复默认设置”“保存并应用”“取消更改”；保存时只提交发生变化的设置分组，不额外触发无关配置写入。
- 控制面板通过托盘按需打开；关闭窗口会结束当前面板会话，因此未保存的本地草稿不会跨关闭保留。
- 页面底部应提供“重新查看新手引导”入口，允许用户在首次引导完成或跳过后再次打开轻量上手流程。

## 3. 通用设置

### 3.1 模块定位

通用设置承载的是影响最广、使用最频繁的基础配置。它既作用于控制面板本身，也会影响仪表盘、通知表现、下载路径和部分系统级体验，因此应作为默认首屏。

### 3.2 语言设置

#### 3.2.1 设置目标

用于统一控制 **仪表盘** 与 **控制面板** 的界面语言。

#### 3.2.2 可选项

- 简体中文
- English

#### 3.2.3 交互规则

- 切换语言后，应明确提示“该设置在重启后生效”。
- 若用户当前存在未完成任务，应提示：语言切换不会中断任务执行，但界面文案更新需在重启后完成。
- 若切换后未立即重启，应在页面顶部保留“待重启以应用语言设置”的轻提示。

#### 3.2.4 页面说明文案建议

> 更改后将同时影响仪表盘与控制面板的界面语言，需重启应用后生效。

### 3.3 开机自启

#### 3.3.1 设置目标

控制 CialloClaw 是否随系统启动自动运行。

#### 3.3.2 默认状态

- 默认开启

#### 3.3.3 交互规则

- 开启时，应说明应用会在系统启动后自动进入后台或按上次状态恢复。
- 关闭时，应说明不影响当前运行，只影响下次开机行为。

### 3.4 外观模式

#### 3.4.1 设置目标

用于调整控制面板整体视觉主题。

#### 3.4.2 可选项

- 跟随系统
- 浅色模式
- 深色模式

#### 3.4.3 交互规则

- 该设置应即时预览。
- 当用户选择“跟随系统”时，应以辅助文案说明：界面会根据系统主题自动切换。
- 若仪表盘与控制面板需要保持统一主题，应明确说明其联动关系。

### 3.5 通知与声音

#### 3.5.1 设置目标

用于调整控制面板与悬浮球相关的声音反馈行为。

#### 3.5.2 建议拆分的设置项

1. **声音总开关**
   - 控制是否播放任务成功、提醒到达、悬浮球气泡提示等音效或语音提示。
2. **提示音/语音风格选择**
   - 例如：不同声线版本的“Ciallo”。
3. **典型触发场景说明**
   - 任务成功
   - 便签到期
   - 授权提醒
   - 高价值信息球弹出

#### 3.5.3 交互规则

- 开启声音时，应支持试听。
- 关闭总开关后，风格选择区域应置灰但保留当前已选项。
- 声音设置应明确说明是否影响系统通知、仅影响应用内反馈，还是两者都影响。

### 3.6 下载设置

#### 3.6.1 设置目标

管理下载内容与任务生成文件的本地保存位置。

#### 3.6.2 建议包含的设置项

1. **本地工作区路径**
   - 用于指定默认工作区位置。
2. **下载内容保存位置**
   - 用于指定下载文件默认落盘路径。
3. **下载前是否逐个询问保存位置**
   - 开启后，每次下载前弹出位置确认。
   - 关闭后，自动保存到默认路径。

#### 3.6.3 交互规则

- 路径设置应支持“选择文件夹”“在系统资源管理器中打开”“恢复默认路径”。
- 打包版默认工作区应解析到用户本机 `AppLocalData/CialloClaw/workspace`，而不是 repo-relative `workspace` 占位目录。
- 宿主侧本地打开 / 定位路径应只信任当前运行时 `workspace root` 与明确白名单的 runtime 子目录（当前仅接受 `temp/...`）；任务源便签解析应只信任当前运行时 `workspace root` 或宿主 `runtime root`，不能回退到编译机 repo 路径。
- 修改路径后，应说明仅影响后续文件，不迁移历史内容。
- 修改工作区路径后，应明确提示“需要重启后生效”；当前任务不热切换到底层 workspace runtime。
- 若目标路径无写权限或不可用，应即时提示并阻止保存。

## 4. 悬浮球

### 4.1 模块定位

该分组用于配置悬浮球作为桌面端常驻入口的表现方式。它决定 Agent 在桌面上的“在场感”强弱，以及用户进入仪表盘与控制面板的便捷程度。

### 4.2 悬浮球设置

#### 4.2.1 自动贴边

用于控制悬浮球在停止拖拽后是否自动吸附到屏幕边缘。

- 开启：适合减少遮挡，保持桌面整洁。
- 关闭：保留用户放置位置。

#### 4.2.2 空闲时半透明

用于控制悬浮球在无操作状态下是否降低不透明度。

- 开启：降低视觉打扰。
- 关闭：持续保持完整可见。

#### 4.2.3 固定位置 / 可拖拽

用于控制悬浮球是否允许用户手动移动。

- 固定位置：适合希望桌面入口稳定不变的用户。
- 可拖拽：适合多屏、多窗口协作场景。

#### 4.2.4 大小

用于控制悬浮球尺寸。

建议提供：

- 小
- 中（默认）
- 大
- 自定义滑杆

### 4.3 页面表达建议

- 悬浮球相关设置最好附带预览区域，让用户即时看到尺寸、透明度与贴边效果。
- 当选择“固定位置”时，位置调整入口应隐藏或置灰。
- 当选择“可拖拽”时，可提供“重置到默认位置”按钮。

### 4.4 设计原则

悬浮球设置的重点不是“炫”，而是让用户在不同工作状态下决定 Agent 是更显眼、更主动，还是更安静、更不打扰。其状态语言应尽量直白，例如“减少遮挡”“更易发现”“降低打扰”。

## 5. 记忆

### 5.1 模块定位

记忆设置用于控制镜子模块如何记录、保留、总结与更新对用户的长期理解。它不应被表现为抽象的 AI 参数，而应被解释为一种协作连续性机制。

### 5.2 镜子记忆开关

#### 5.2.1 设置目标

控制是否启用镜子记忆能力。

#### 5.2.2 说明文案建议

> 帮助 CialloClaw 逐步理解你的偏好、任务状态与工作方式，让协作更连续。

#### 5.2.3 交互规则

- 关闭后，应明确提示：未来的新记忆将不再记录，但已保存内容如何处理需要额外说明。
- 关闭时，建议弹出二次选择：
  - 仅停止后续记录
  - 同时清理现有镜子记忆

### 5.3 生命周期

#### 5.3.1 设置目标

控制镜子记忆的保留时长。

#### 5.3.2 可选项

- 仅本轮会话
- 保留 7 天
- 保留 30 天
- 长期保留

#### 5.3.3 交互规则

- 切换保留周期时，应说明新策略对“已有记忆”和“新增记忆”分别如何生效。
- 若从长期保留改为短期保留，应说明旧数据是否按新规则清理。

### 5.4 周期总结

#### 5.4.1 设置目标

控制系统自动生成工作总结与偏好画像的频率。

#### 5.4.2 建议拆分的设置项

1. **每 X 天 / 周生成一次工作总结**
2. **每 X 周更新一次偏好画像**

#### 5.4.3 页面说明建议

- 工作总结偏向“这段时间做了什么”。
- 偏好画像偏向“你通常怎么工作、偏好怎样被协助”。

#### 5.4.4 交互规则

- 应允许用户关闭自动总结，仅保留手动生成。
- 若开启自动总结，应说明总结结果将出现在哪个模块中，例如镜子页或日报入口。

### 5.5 记忆管理建议补充

虽然原始需求只列出开关与生命周期，但为了形成闭环，建议在记忆设置页补充以下能力：

- 查看最近一次总结生成时间
- 查看当前保留中的记忆数量
- 快速进入“记忆管理”页
- 删除指定类别记忆
- 忽略某一类洞察建议

这样用户对“记忆正在发生什么”会更有掌控感。

## 6. 任务与巡检

### 6.1 模块定位

该分组负责定义 Agent 如何发现待办、以什么节奏巡检、在什么时机主动提醒，以及何时将未来安排转化为实际推进的任务。它对应的是从“便签协作”到“任务状态”的自动化协作链路。

### 6.2 待办来源

#### 6.2.1 设置目标

指定控制面板用于识别待办、事项或任务素材的来源位置。

#### 6.2.2 当前需求

- 指定任务文件夹

#### 6.2.3 交互规则

- 应支持选择本地文件夹。
- 应支持查看当前已绑定路径。
- 修改后应说明：后续巡检基于新路径执行，不自动迁移旧规则。
- 设置页只负责维护“任务来源列表”本身；真正的新建 markdown 便签与内容编辑，应回到“便签协作”页完成。

### 6.3 巡检规则

#### 6.3.1 巡检频率

用于控制系统定时扫描待办来源的时间间隔。

可选表达：

- 每 X 分钟
- 每 X 小时
- 每 X 天

#### 6.3.2 文件变更时立即巡检

用于控制指定文件夹中的内容一旦发生变更，是否立即触发检查。

#### 6.3.3 启动时巡检

用于控制应用启动后是否立即执行一次巡检。

设置页在这里定义“何时巡检”，但不承担手动巡检按钮；手动触发入口应放在“便签协作”页，方便用户在修改源便签后立即刷新识别结果。

### 6.4 提醒机制

#### 6.4.1 截止前提醒

当事项接近截止时间时，主动提醒用户。

#### 6.4.2 长时间未处理提醒

当事项长时间处于未完成状态时，主动提醒用户。

### 6.5 页面组织建议

可按“来源—巡检—提醒”三层组织：

1. **来源配置**：我从哪里获取待办。
2. **巡检策略**：我多久检查一次、是否在变更时检查。
3. **提醒策略**：发现临期或积压后如何提示。

“来源配置”不应直接演变成编辑器；它负责告诉系统去哪里读文件。具体的 markdown 新建、保存和查看，应由“便签协作”页承接，以保持设置页聚焦于策略而不是内容创作。

### 6.6 关键交互规则

- 当巡检频率较高时，应在说明文案中提示可能带来的资源消耗变化。
- 当关闭所有提醒后，应提示：系统仍会巡检，但不再主动打断用户。
- 当待办来源无效时，应在设置页直接暴露异常状态，而不是等到任务页才发现。

## 7. 模型与安全（当前正式导航）

### 7.1 模块定位

该分组承载模型路由、预算策略、API Key 状态与安全摘要，是用户建立信任、控制成本、理解资源消耗的关键入口。它与安全卫士中的待授权数量、恢复点、Token/费用总览保持一致。

### 7.2 成本统计

#### 7.2.1 建议展示的信息

- 当前使用模型
- 模型提供商
- Provider Base URL（当桌面端处于本地兼容模式时）
- Token 消耗统计
- 单任务成本
- 达到预算时自动降级

#### 7.2.1.1 桌面端模型与路由兼容层

当前桌面端控制面板中的“模型与路由”需要同时承接两类数据：

- 正式共享字段：`settings.models.provider`
- 正式凭证字段：`settings.models.credentials.budget_auto_downgrade`、`settings.models.credentials.provider_api_key_configured`、`settings.models.credentials.base_url`、`settings.models.credentials.model`

约束如下：

- `provider`、`budget_auto_downgrade`、`provider_api_key_configured`、`base_url` 与 `model` 都以正式 `models / models.credentials` 为准，不允许再以旧的 `data_log` 兼容字段反向覆盖。
- `API Key` 仍只返回是否已配置的布尔状态，不回传明文。
- 控制面板读取 RPC 设置后，应直接以正式 `models` 结构渲染，不再把共享字段镜像到旧的 `data_log` 兼容视图。

#### 7.2.2 页面表达建议

建议拆成两个层级：

**概览层**

- 当前模型
- 当前提供商
- 今日累计 Token
- 今日累计成本

**策略层**

- 单任务预算阈值
- 每日预算阈值
- 达到预算时自动降级
- 达到预算时是否暂停高成本任务

#### 7.2.3 交互规则

- 当开启“达到预算时自动降级”后，应明确说明会降到哪类模型或策略。
- 当预算已接近阈值时，设置页应出现预警提示，而不是仅在执行时提示。
- 当显示 `Base URL` / `Model` 时，应明确这是桌面端路由兼容视图；如果当前后端协议未提供对应字段，不要把它们表达成已写入正式后端设置的共享配置。
- `API Key` 的保存提示应明确正式边界：
  - 通过 JSON-RPC `agent.settings.update` 提交，只写入后端 Stronghold，不回显明文。
  - 当 RPC 不可用时，控制面板应直接暴露加载或保存失败状态，而不是回退到本地 mock 快照。
- 控制面板应提供“测试连接”入口，并在点击“保存设置”前对当前模型草稿执行只读校验：
  - 若校验失败，应阻止本次保存，并显示可操作的失败原因，例如 API Key 缺失、鉴权失败、Base URL 不存在、模型名错误或工具调用不兼容。
  - 若校验通过，再执行正式 `agent.settings.update` 保存，并保留“已保存 / 下一个任务周期生效”的提示。
  - “测试连接”与保存前校验都不得修改正式设置快照、Stronghold 密钥或当前任务状态。

## 8. 数据与日志（预留展示）

### 8.1 当前状态

当前桌面控制面板尚未把“本地存储位置、删除全部记忆、删除任务历史、恢复默认设置、日志查看”做成独立可操作卡片；这些能力仍属于后续扩展项，但在设计上需要继续与安全卫士和数据真源保持一致。

### 8.2 需求项保留

- 本地存储位置
- 删除全部记忆
- 删除任务历史
- 恢复默认设置

### 8.3 建议拆分的动作层级

1. **查看与定位**
   - 查看本地存储位置
   - 在系统中打开该目录
2. **清理动作**
   - 删除全部记忆
   - 删除任务历史
3. **全局重置**
   - 恢复默认设置

### 8.4 风险分级建议

- 查看本地存储位置：低风险，可直接执行。
- 删除任务历史：中高风险，需要二次确认。
- 删除全部记忆：中高风险，需要明确后果说明。
- 恢复默认设置：高风险，需要说明哪些配置会丢失、哪些数据不会被删除。

### 8.5 二次确认文案建议

删除、清理、恢复类动作都应说明：

- 将删除什么。
- 不会删除什么。
- 是否可恢复。

例如：

> 删除任务历史后，历史任务记录将无法在任务状态页继续查看，但不会删除已输出到本地工作区的文件。

### 8.6 日志能力建议补充

虽然当前正式导航已收口为“模型与安全”，但为了与仪表盘和安全卫士联动，后续仍建议补充以下查看入口：

- 执行日志查看
- 错误日志查看
- Token 日志查看
- 费用日志查看

这样设置页不仅能“改策略”，也能“查原因”。

## 9. 关于

### 9.1 模块定位

关于页承载产品支持、传播与版本管理信息。当前桌面端已经将它收口为正式导航分组，但页面内容仍以帮助与反馈、分享入口和版本信息为主；检查更新等扩展动作继续保留为后续能力。

### 9.2 建议包含的内容

当前桌面端至少应包含：

- 帮助与反馈
- 分享 CialloClaw 给好友
- 当前版本号

后续可继续补充：

- 检查更新

### 9.3 交互规则

- 版本号应明确到可用于排查问题的粒度。
- 当前实现中的反馈与分享按钮采用复制优先策略；若运行环境不支持剪贴板，应直接显示可手动处理的链接文本，避免静默失败。
- 检查更新后，应反馈当前状态：已是最新版本 / 发现新版本 / 更新失败。
- 帮助反馈入口应说明跳转到哪里，例如文档、表单、社区或邮箱。

## 10. 页面层级与交互规范

### 10.1 设置项的标准结构

每个设置项建议统一包含以下元素：

1. **标题**：用简洁动作语义命名。
2. **说明**：解释该项影响范围。
3. **控件**：开关、单选、下拉、输入框、路径选择器等。
4. **状态提示**：例如“立即生效”“重启后生效”“高风险操作”。
5. **辅助操作**：恢复默认、试听、预览、查看详情等。

### 10.2 生效状态的统一语言

建议统一使用以下表达：

- **立即生效**：切换后马上看到效果。
- **重启后生效**：需重启应用后完成变更。
- **仅影响后续任务**：不会追溯修改当前任务。
- **需要确认**：执行前会弹出确认提示。
- **不可撤回**：删除或清理后无法恢复。

### 10.3 空态、禁用态与异常态

#### 首次引导

首次启动时，桌面端应提供一套约 2 分钟的轻量上手引导。该引导不是帮助中心，也不是完整教程，而是围绕真实界面帮助用户完成最基础的首次使用路径。

- 目标仅限于：认识悬浮球、学会长按语音、学会双击打开主界面、理解主界面切换入口、知道托盘到控制面板的路径、完成 `API Key` 配置。
- 交互形式应以轻量说明卡和少量辅助文案为主，不单独制作厚重教程页。
- 引导展示建议由一个独立的透明引导窗口负责；该窗口不再叠加整屏灰色遮罩，并保持 `topmost`，控制面板本身继续负责真实输入、保存和状态反馈，不在引导层里承接业务操作。
- 每一步都必须允许继续、返回上一步或结束引导，不可因为监听不到真实操作而卡住流程。
- 当用户完成或主动结束后，后续不再自动弹出；但仍可通过控制面板底部入口重新查看。
- 当用户在引导过程中进入控制面板时，应自动收起仪表盘窗口，并在用户通过控制面板底部入口重新查看引导时自动关闭当前控制面板窗口。
- 在 `API Key` 配置步骤中，若用户选择跳过，应明确提示“未配置 API Key 时，部分功能暂不可用”。

#### 空态

当尚未设置路径、未启用记忆、未配置预算时，不应留白，而应提供明确引导，例如：

- 尚未选择任务文件夹
- 尚未开启镜子记忆
- 尚未设置预算阈值

#### 禁用态

当某项能力受上层开关控制时，应置灰并解释原因，例如：

- 已关闭声音提示，因此暂不可选择语音风格。
- 已关闭镜子记忆，因此总结频率设置暂不可用。

#### 异常态

当路径无效、权限不足、模型不可用、更新失败时，应在原位置展示错误信息和建议操作，而不是只用红字提示。

- 当控制面板与正式 RPC 设置服务断开时，不应直接退化为整页空白；应保留设置页主框架，并在页面顶部通过独立错误块提示连接失败、重试入口，以及“当前仅显示上一次成功同步的本地快照”的说明。

## 11. 与仪表盘的联动要求

控制面板设置虽然是独立配置入口，但其结果应明确反馈到仪表盘各模块中：

### 11.1 与首页 / 意识场联动

- 外观模式、语言、通知声音会直接影响首页整体感知。
- 高价值信息球与悬浮球的表现，应受外观与通知设置控制。

### 11.2 与任务状态联动

- 任务文件夹、巡检频率、提醒规则，会影响任务的生成与推进节奏。
- 预算与模型策略，会影响任务执行成本与降级行为。

### 11.3 与镜子联动

- 镜子记忆开关、生命周期、总结频率会直接影响镜子模块中的记忆数量、日报与偏好画像更新节奏。

### 11.4 与安全卫士联动

- 删除数据、恢复默认、预算策略、日志查看入口，应与安全卫士中的风险解释、执行记录、恢复逻辑保持一致。

这样用户才能理解：设置不是孤立配置，而是在持续塑造 Agent 的协作方式。

## 12. 建议采用的分组文案（可直接用于 UI）

### 通用设置

用于管理语言、启动方式、主题、通知与下载位置等基础体验设置。

### 悬浮球

用于调整悬浮球的显示方式、位置与桌面在场感。

### 记忆

用于控制镜子记忆的启用状态、保留周期与自动总结节奏。

### 任务与巡检

用于指定待办来源，并配置巡检、提醒与自动化协作策略。

### 模型与安全

用于查看模型与成本信息，并理解当前安全摘要、预算策略和凭证状态。

### 数据与日志（预留展示）

用于查看模型与成本历史，并管理本地数据、隐私与日志。

### 关于

用于查看版本信息、获取帮助与反馈，并提供分享入口；检查更新继续作为后续扩展能力保留。

---

## 13. 结论

规范化后的控制面板设置，不应只是把现有配置项分类展示，而应形成一套完整、可解释、可追溯的设置体验：

- 通用设置负责高频基础控制。
- 悬浮球负责 Agent 的在场方式。
- 记忆负责长期协作连续性。
- 任务与巡检负责从待办到执行的转化机制。
- 模型与安全负责成本、边界与信任管理。
- 数据与日志、日志查看与数据清理等展示仍属于后续扩展项，但应继续保留在文档里，进入正式导航前只需要明确标注其状态，而不是从设计文档中移除。

在此基础上，设置页应与仪表盘首页、任务状态、镜子和安全卫士共享统一语言，让用户理解每一次配置调整都会如何影响 Agent 的实际协作行为。
