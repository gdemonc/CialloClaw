# AGENTS.md

本文件只补充 `services/local-service` 及其子目录的后端专项规则。

进入本目录前，必须先读根目录 `AGENTS.md`。
本文件不重复共享铁律，只补充后端专属边界、阅读路径、实现顺序、Go 风格约束与注释要求。

------

## 1. 目录范围

本文件覆盖：

- `services/local-service/cmd`
- `services/local-service/internal`

若当前任务会同时修改协议真源、前端承接方式、共享 schema 或跨端时序，必须回到根目录 `AGENTS.md`，并补读相关上位文档，不得仅凭本文件决策。

------

## 2. 执行优先级与例外边界

本文件的规则不是同一优先级。
当规则之间发生冲突时，按以下顺序裁决：

1. 先保正式协议、状态语义、错误码、交付链路与风险治理边界
2. 再保行为正确性、兼容性、恢复能力与可审计性
3. 再做类型收紧、文件拆分、命名优化、文案收口与风格清理
4. 性能、扩展性与审美优化排在以上事项之后

必须坚持：

- 修 bug、补兼容、做热修时，先控制改动面，避免顺手扩大为结构重构
- 若结构优化会放大回归面或影响发布节奏，可拆到后续独立工作，不得挟带进入高风险修复
- 当风格优化与真源契约冲突时，永远以真源契约为先

允许存在受控例外，但必须显式说明原因、范围和后续计划：

- 紧急 hotfix 或线上兼容修复
- 迁移中的兼容 shim、过渡 DTO、双写/双读路径
- 第三方透传 payload、插件原始结果、审计扩展 blob
- 生成代码、镜像代码或外部接口适配层
- 尚未稳定、且明确标记为临时的实验分支逻辑

出现例外时，至少补充：

- 为什么当前不能按理想结构落地
- 临时边界在哪里
- 后续清理条件或退出条件是什么

## 3. 后端文档路由

开始修改前，至少先回答：

1. 当前改动属于哪个后端域：rpc、orchestrator、context、intent、delivery、risk、memory、storage、runengine，还是其他服务？
2. 这次改动影响的是 `task` 对外语义、`run` 内部执行语义，还是两者映射关系？
3. 是否新增或改变了正式协议对象、状态、错误码或数据字段？
4. 是否涉及工具调用、授权、审计、恢复点或正式交付出口？
5. 工具结果会如何回流到 `tool_call / event / delivery_result` 链？

若任一问题回答不清，优先按下面路由补读：

| 任务类型 | 应优先读取的文档 |
| --- | --- |
| 编排、状态机、服务职责、跨模块时序 | `docs/module-design.md`、`docs/work-priority-plan.md` |
| 协议对象、JSON-RPC 方法、通知、错误码 | `docs/development-guidelines.md`、`docs/protocol-design.md`，必要时核对 `packages/protocol` |
| 表结构、字段、索引、运行态 / 记忆 / 治理分层 | `docs/development-guidelines.md`、`docs/data-design.md` |
| 主链路、统一边界、跨端职责不清 | `docs/architecture-overview.md`、`docs/development-guidelines.md`、`docs/work-priority-plan.md` |

禁止：

- 跳过真源直接改编排或状态机
- 只看局部实现不核对对象语义
- 以“先跑通”为理由绕过交付、治理或协议边界

## 4. 后端边界

后端负责：

- JSON-RPC Server
- Orchestrator
- Context Manager
- Intent / Planning
- RunEngine / Task 状态机
- Delivery
- Memory
- Risk / Audit / Recovery
- Storage
- Trace / Eval
- Tool / Plugin / Sandbox / Adapter

后端不得：

- 感知前端页面组件树
- 依赖 Tauri 细节
- 输出未登记对象给前端长期消费
- 绕过正式交付出口直接把工具原始结果当最终结果

必须坚持：

- 保持 `task` 与 `run` 稳定映射
- 让工具结果先进入 `tool_call / event / delivery_result` 链
- 让风险动作可授权、可审计、可恢复
- 让错误进入正式错误码体系

禁止：

- 绕过编排器直连前端
- 绕过交付内核直接回结果
- 把记忆逻辑侵入运行态状态机
- 把平台细节写死在业务层
- 让 intent 退化成固定关键词分类器并替代 Agent Loop 主执行路径

------

## 5. 后端实现顺序

实现顺序必须优先满足：

1. 保住 `task-centric` 主链路和 `run` 执行兼容链
2. 保住 JSON-RPC 契约、错误码和数据真源的一致性
3. 保住工具调用、风险治理、交付出口和事件回流的完整性
4. 在不破坏以上三项的前提下，再做类型收紧、结构拆分、命名优化和外围增强

如果某个实现需要临时绕开风险、交付或正式对象边界才能跑通，该实现不能进入正式分支。

------

## 6. 后端改动前后必做事项

修改前至少确认：

- 本次改动触及的是协议、编排、存储、治理还是交付边界
- `task` 与 `run` 的映射是否会受影响
- 是否会影响 `tool_call / event / delivery_result` 回流
- 是否需要同步协议文档、数据文档、模块文档或排期文档

修改后至少自检：

- `task` 与 `run` 的映射是否保持稳定
- 是否新增了协议 / 状态 / 错误码 / 字段；若有，是否已同步真源与文档
- 工具结果是否正确回流到 `tool_call / event / delivery_result`
- 风险动作是否经过授权、审计和恢复点链路
- 当前改动是否补齐所需英文注释
- 是否补到了对应的测试或失败路径验证

## 7. 后端文档回写规则

以下情况发生时，必须同步回写相关文档：

- 调整编排职责、状态机、跨模块时序或主链路收口方式
- 新增或调整正式协议对象、方法、状态、错误码
- 新增或调整核心表、字段、索引、运行态 / 记忆 / 治理分层
- 调整风险、授权、审计、恢复点或正式交付出口的语义
- 代码已真实完成 `docs/work-priority-plan.md` 中的对应任务项

回写时遵循：

- 协议变化优先回写协议真源和开发规范文档
- 数据变化优先回写数据设计文档
- 模块职责和编排变化优先回写模块设计文档
- 若当前优先级任务已经完成，不要让排期文档长期滞后于代码

## 8. 后端注释规范

注释在后端不是可选项，但也不能泛滥。
尤其以下场景必须写英文注释：

- 状态机迁移、确认语义、重试与恢复逻辑
- 工具调用回流、事件生成、交付对象构建
- 授权、审计、恢复点、风险分级相关逻辑
- 并发控制、锁、队列、超时、重放、幂等处理
- 存储映射、字段语义、兼容边界和错误包装
- 兼容 shim、迁移逻辑或临时例外边界

注释至少要说明：

- 这段逻辑在主链路中的职责
- 为什么当前状态迁移、错误包装或回流顺序必须这样设计
- 风险、恢复或授权边界在哪里
- 失败路径、兼容边界和幂等假设是什么

必须坚持：

- 新增复杂逻辑时同步写英文注释
- 调整复杂逻辑时同步检查并修正旧注释
- 当前改动范围内若发现中文注释，必须改成英文注释
- 顶层声明的英文 doc comment 必须说明 contract、invariant、side effect 或限制条件，不得只把声明内容重复一遍
- 新增或调整正式协议方法、正式通知、正式字段、核心运行态结构或排期勾选时，必须在同一次工作中同步回写对应真源文档

优先做法：

- 只在非显然边界、跨阶段约束、恢复路径、兼容逻辑和风险点处写解释型注释
- 简单私有 helper 若逻辑直白，可不补解释型注释
- 用注释解释“为什么必须这样”，不要把代码表面含义翻译一遍

禁止：

- 用“函数名已经很清楚”作为不写注释的理由
- 让关键失败路径、恢复路径和授权路径缺少解释
- 让注释与实际状态迁移、错误处理或数据语义失真
- 使用“定义当前模块结构”“处理当前模块逻辑”一类无信息密度的空话注释
- 用大段注释淹没本来已经清晰的简单实现

## 9. Go 代码风格与结构约束

本仓库后端 Go 代码以以下目标为准：

- 优先让类型系统承担协议约束与重构保护
- 优先让成功路径向下流动，减少嵌套和分支缠绕
- 优先让文件按职责与阶段分组，而不是持续膨胀成上帝文件
- 优先让注释解释边界、失败路径和兼容原因，而不是重复语法表面含义

### 9.1 类型边界与协议真源

必须坚持：

- 涉及正式 JSON-RPC 方法、通知、错误对象、交付对象时，必须先对齐 `packages/protocol` 与相关文档真源
- `rpc` 层负责 decode / validate / adapt；穿过 `rpc` 边界后，正式入参与出参应优先使用 typed request / response structs
- 所有正式协议 struct 必须补齐清晰的 JSON tag；字段可选性必须显式表达
- `map[string]any` 不应作为默认正式对象；它只应停留在真正动态、无法静态建模或明确标记为过渡层的边界

优先做法：

- 为高频方法优先建立 typed DTO，例如 `task.start`、`input.submit`、`task.detail.get`
- 把 response 组装收口到明确的 builder 或 DTO 转换函数中
- 让字段名、错误码、状态值可被类型、常量和跳转引用追踪

允许但必须说明原因的例外：

- 插件透传、第三方原始 payload、审计扩展 blob
- 迁移期的兼容 envelope、过渡对象或 adapter 层
- 实验性字段尚未进入正式真源前的临时隔离区

禁止：

- 在 `rpc` 之外把 `map[string]any` 当成默认正式入参或出参
- 用字符串 key 在业务层长期手搓正式协议对象
- 在不知道真源字段语义的情况下直接扩充 JSON shape

### 9.2 文件与函数规模

必须坚持：

- 单个文件应只承载一组紧密相关的职责
- 单个函数应只完成一个主要阶段，避免同时承担校验、编排、状态迁移、文案拼装和协议组装

建议阈值：

- 单函数以 80 行左右为重构信号；超过后优先判断是否应拆阶段或抽 builder / helper，而不是机械强拆
- 单文件以 400 行左右为重构信号；超过后优先按职责或阶段拆分，而不是把阈值当成提交阻断条件
- 结构体构造、运行时通知、任务启动、输入提交、查询接口、错误映射、response builder 不应持续堆叠在同一文件

优先拆分方式：

- 按行为阶段拆：如 `task_start.go`、`input_submit.go`、`task_query.go`、`task_control.go`
- 按支撑职责拆：如 `response_builders.go`、`error_specs.go`、`messages.go`
- `service.go` 只保留核心结构、构造函数与少量总入口，不承载所有细节实现

禁止：

- 把新行为持续追加到已有巨型文件中
- 因为“先放一起更快”而让局部性继续恶化
- 为了满足行数阈值而无意义切碎本来紧密相关的逻辑
- 把测试也写成对应的万能大文件

### 9.3 控制流、嵌套与成功路径

必须坚持：

- 成功路径向下流动，guard clause、错误路径和特殊路径尽早 return
- 优先减少多层嵌套、长 `else if handled` 链和跨阶段条件缠绕
- response 组装与业务判断尽量分离，避免在主流程中反复插入协议拼装细节

优先做法：

- 先处理 invalid / noop / special case，再进入主流程
- 用阶段函数显式表达流程，例如 `prepare*`、`maybe*`、`build*`
- 将复杂状态迁移和失败包装拆入具名 helper，并补英文注释解释不变量

禁止：

- 把治理、执行、展示、交付、气泡文案和状态更新交错堆在一个函数中
- 明明可以 early return 还持续包裹 `else`
- 让读者必须在同一屏中同时记住多个前置条件与多个返回形状

### 9.4 构造器与依赖装配

必须坚持：

- 明确区分 required dependencies 与 optional capabilities
- 必选依赖优先集中到 `Deps` 结构或清晰的 staged build 中
- 可选能力优先使用明确的 option 或后置配置点，但不得造成半初始化对象长期暴露

优先做法：

- 把 bootstrap 装配拆成 `buildCoreDeps`、`buildRuntimes`、`buildServices` 一类阶段函数
- 让 `New(...)` 明确表达创建后对象的最小可用状态
- 测试中允许替换 fake 依赖，但替换入口必须清晰可见

禁止：

- 长参数构造与多个 `WithXxx` 后置变异混搭成一锅炖
- 让读者无法判断哪些依赖缺失会破坏构造后对象的基本语义
- 在构造过程中穿插与核心依赖无关的展示或文案逻辑

### 9.5 包命名、import 与 alias

必须坚持：

- 包名短、准、自然，默认 import 后应可直接表达语义
- 高频业务包命名应尽量避免与标准库高频包长期冲突

优先做法：

- 若某个业务包需要长期以 `contextsvc` 一类 alias 才能避开 stdlib，优先评估是否应重命名包
- 评估包重命名时，必须同时考虑迁移成本、历史引用、跨模块影响和收益范围
- alias 仅用于必要冲突消解或明显提升可读性的场景

禁止：

- 让整个项目长期背负不必要的 alias 税
- 用模糊、过泛的包名增加阅读歧义
- 在没有明显收益时贸然发起大面积重命名

### 9.6 错误映射、用户文案与交付组装

必须坚持：

- 错误 contract 应集中定义；当同类映射已经跨多个 handler 或流程复用时，优先改为声明式映射，而不是长期散落在长 `if errors.Is(...)` 链中
- 用户可见文案、标题、展示 copy 应尽量从编排主流程中收口出去
- `orchestrator` 返回语义与状态；最终文案与展示对象优先由 `delivery`、message registry 或专门 builder 统一组装

优先做法：

- 当错误映射或文案已经跨模块复用时，使用 `error_specs.go`、registry、表驱动映射或 `messages.go` 一类集中层维护
- 让屏幕分析、审批、治理结果等混合对象尽早变成 typed state object
- 在小范围或尚未稳定的逻辑里，可先保持局部实现，但不得继续无约束扩散

禁止：

- 在 handler 或 orchestrator 主流程中无节制散落协议错误壳构造
- 在编排层长期直接埋 UI 文案并耦合展示表达
- 一边做状态迁移一边拼接最终用户可见 copy
- 为了“集中化”而引入明显重于问题本身的抽象层

### 9.7 测试组织与验证方式

必须坚持：

- 测试按行为组织，不按“所有场景塞进一个文件”组织
- 矩阵型输入输出优先 table-driven tests
- 流式通知、长链路、并发与恢复场景优先拆成窄而清晰的行为测试

优先做法：

- 为错误映射、参数校验、dispatch matrix、transport 变体使用表驱动测试
- 为运行时通知顺序、恢复点、治理链路使用单独测试文件与清晰命名
- 测试数据、假对象和 helper 也要按职责收口，避免隐藏关键前提

本地最小检查建议：

- `goimports`
- 受影响包的 `go test`

提交或 PR 前完整检查建议：

- `go test ./...`
- `go vet ./...`
- `staticcheck ./...`

若当前环境无法完整执行，必须明确说明哪些未执行、为什么未执行、残留风险在哪里。

### 9.8 入口与退出策略

必须坚持：

- `cmd` 层尽量保持 boring，入口函数只做最薄的 wiring 与退出管理
- 主要逻辑优先下沉到可测试的 `run() error` 或等价入口函数中

禁止：

- 在 `main` 中堆叠大量构造、分支与副作用
- 在多个位置随意 `Fatal` 导致退出策略分散

## 10. CodingAgent 执行补充要求

当 CodingAgent 修改 `services/local-service` 后端代码时，回复中需要补充自检摘要，但应按改动规模分级。

### 10.1 小改动

适用：

- 局部 bugfix
- 单字段修正
- 小范围注释修复
- 不改变正式协议与主链路语义的小型补丁

至少说明：

1. 本次改动触及了哪些后端域
2. 是否改动了正式字段、状态、错误码、通知或 typed DTO
3. 执行了哪些测试与检查；未执行项及原因是什么

### 10.2 大改动

适用：

- 新增或调整正式协议对象、状态、错误码、通知
- 拆分巨型函数、巨型文件或测试结构
- 调整编排、交付、治理、恢复点或工具回流主链路
- 引入新的 typed DTO、builder、集中化错误映射或文案层

必须补充：

1. 本次改动触及了哪些后端域
2. 是否新增或修改了 typed DTO、正式字段、错误码、状态或通知
3. 是否拆分了巨型函数或巨型文件；拆分后新文件分别承载什么职责
4. 是否修正或新增了英文注释；覆盖了哪些关键失败路径或边界说明
5. 是否把错误映射、用户文案或 response builder 做了集中化处理
6. 执行了哪些测试与检查；未执行项及原因是什么

如果本次改动仍保留以下技术债，且这些技术债与本次改动直接相关，必须主动说明：

- 仍存在的正式 `map[string]any` 边界

## 11. 建议的后端工作模式

5. 在不扩大风险面的前提下，优先收紧类型边界、拆小文件、拉直控制流
6. 改完后补注释、测试、自检并回写相关文档
